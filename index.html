기존 문제

AI는 있었지만, 발화에서 응답까지 흐르는 길이 없었다

각 서비스는 개별 개발 상태. 입력은 저장까지만 도달하고, 발화·센서 신호가 음성 응답으로 닿지 못하는 단절 구간을 이어야 했습니다.

  • API · LLM · 음성 · 프론트엔드가 개별 개발 상태로 분리
  • RP2040 마이크 입력을 지속 처리할 주체 부재
  • Wemos 센서 데이터는 DB 저장까지만, 이후 동작 연결 없음
  • 발화·거리 감지가 실제 음성 응답으로 이어지지 않음
맡은 역할

  1. STT · LLM · TTS 음성 파이프라인 설계

  3. 거리 센서 기반 선제 인사 이벤트 연결
  4. end-to-end 호출 경로 구성
시스템 구조

입력 3개 → voice-agent → STT · LLM · TTS → 음성 재생

3개 입력 경로가 voice-agent 오케스트레이션으로 수렴하고, LLM은 대화 답변과 장치 동작 제안을 동시에 출력합니다. 제안은 검증 계층을 거쳐 안전한 범위로만 강제됩니다.

LLM 응답 계약 설계

03

허용 설정 검증 계층

05

센서 → AI 트리거 연결

Wemos 거리 센서 입력을 LLM 기반 선제 인사 트리거로 연결. 사람이 다가오면 먼저 말을 거는 흐름을 만들었습니다.

llm_response.json 
// LLM 단일 응답 = 대화 + 동작 제안
{
  "answer": "네, 거실 조명을 켤게요.",
  "action_proposal": {
    "setting": "wake_enabled",
    "value": true
  }
}

// → 검증 계층(allow-list) 통과 후
//   실제 장치에 반영

🛡 허용 설정 검증 계층

LLM의 자유로운 제안을 그대로 실행하지 않고, 사전 정의된 안전 범위 안에서만 적용합니다.
