# Super Marimo STAR

## 사례 1. 분리된 서비스를 음성·센서 기반 AIoT 흐름으로 연결

**적합 문항**: 문제 해결, 시스템 통합, 협업, AIoT·백엔드 직무 역량

### Situation

- API 서버, LLM 서버, 음성 처리 서버, 프론트엔드의 개별 개발 상태
- RP2040 마이크 입력을 지속적으로 처리할 주체의 부재
- Wemos 센서 데이터의 DB 저장 이후 동작 연결 부재
- 사용자 발화와 거리 감지가 실제 음성 응답으로 이어지지 않는 상태

### Task

- 사용자 발화를 STT·LLM·TTS로 잇는 음성 대화 파이프라인 설계
- LLM 응답이 단순 답변을 넘어 장치 동작을 제안하도록 응답 계약 정의
- 거리 조건 충족 시 LLM 기반 선제 인사 이벤트 생성 및 재생 연결
- 위 AI 흐름을 담는 end-to-end 호출 경로 구성 (서비스 간 통합 인터페이스는 최소 규격으로 정리)

### Action

#### 직접 수행 (AI 흐름 중심)

- `voice-agent` 오케스트레이션 설계: STT → LLM → TTS 단계 조율 및 상태 관리
- LLM 응답 계약 설계: `answer`(대화) + `action_proposal`(장치 제어 제안) 이중 출력 구조
- LLM 제안에 대한 허용 설정 검증 계층 구현 (안전한 장치 동작 범위 강제)
- Windows Bridge의 RP2040 PCM 수신·음성 구간 감지(VAD)·재생 제어 구현
- Wemos 거리 센서 입력을 LLM 기반 선제 인사 트리거로 연결

#### 협업·통합 (간략)

- 팀원 API·LLM·음성·프론트엔드 구성요소의 입출력 규격 확인 및 통합 경로 조정
- 병렬 개발 결과의 `main` 반영, Docker Compose 실행 순서·프로필 정리
- 팀 전체 결과물 기준 동작 흐름 및 운영 문서 정리

#### 도구 활용

- Git 로그·커밋 이력을 통한 변경 범위와 작성 근거 확인
- Docker Compose를 통한 다중 서비스 기동 순서와 연결 상태 확인
- pytest를 통한 API·Bridge·voice-agent·LLM 계약 테스트 구성
- 시리얼 로그와 서비스 로그를 통한 STT·LLM·TTS 단계별 흐름 점검

### Result

- 실제 음성 응답으로 연결된 통합 입력 경로: **0개 → 3개**
  - RP2040 마이크 기반 음성 입력
  - Wemos 거리 센서 기반 선제 반응
  - 프론트엔드 기반 수동 입력
- LLM이 변경을 제안할 수 있는 런타임 설정 항목: **0개 → 3개**
  - `wake_enabled`
  - `distance_detection_enabled`
  - `distance_threshold_cm`
- 센서값 저장 중심 구조에서 센서 이벤트·LLM 응답·TTS 재생이 연결된 구조로 전환
- Git 기본 작성자 기준 커밋: **전체 61개 중 37개**
  - 커밋 수는 작업량 참고 지표이며 기여 품질이나 기여도 비율로 단정하지 않음

### 면접 한 줄

개별 개발된 API·LLM·음성·센서 서비스를 연결해, 사용자 발화와 거리 감지가 실제 음성 응답으로 이어지는 3개 입력 경로의 AIoT 흐름 구성.
