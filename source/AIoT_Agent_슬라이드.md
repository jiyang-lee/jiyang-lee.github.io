# Super Marimo
## 분리된 서비스를 음성·센서 기반 AIoT 흐름으로 연결

STT · LLM · TTS 음성 대화 + 거리 센서 선제 반응

<!-- 적합 문항: 문제 해결 / 시스템 통합 / 협업 / AIoT·백엔드 직무 -->

---

# Situation — 무엇이 문제였나

- API · LLM · 음성 처리 · 프론트엔드가 **개별 개발 상태**로 분리
- RP2040 마이크 입력을 **지속 처리할 주체 부재**
- Wemos 센서 데이터는 **DB 저장까지만**, 이후 동작 연결 없음
- 사용자 발화·거리 감지가 **실제 음성 응답으로 이어지지 않음**

> 핵심: AI는 있지만, 발화에서 응답까지 흐르는 길이 없었다

---

# Task — 무엇을 해내야 했나

**AI 음성 대화 흐름 완성에 초점**

1. 사용자 발화를 **STT → LLM → TTS** 로 잇는 음성 파이프라인 설계
2. LLM이 단순 답변을 넘어 **장치 동작까지 제안**하는 응답 계약 정의
3. 거리 조건 충족 시 **LLM 기반 선제 인사** 이벤트 생성·재생
4. 위 AI 흐름을 담는 end-to-end 호출 경로 구성
   *(서비스 간 통합 인터페이스는 최소 규격으로 정리)*

---

# Action ① — AI 흐름 직접 구현

- **`voice-agent` 오케스트레이션 설계**
  STT → LLM → TTS 단계 조율 및 상태 관리
- **LLM 응답 계약 설계**
  `answer`(대화) + `action_proposal`(장치 제어 제안) 이중 출력
- **허용 설정 검증 계층**
  LLM 제안을 안전한 장치 동작 범위로 강제
- **장치 입력 처리**
  RP2040 PCM 수신 · 음성 구간 감지(VAD) · 재생 제어
- **센서 → AI 트리거 연결**
  Wemos 거리 센서를 LLM 선제 인사 트리거로 연결

---

# Action ② — 협업·통합 & 도구

**협업·통합 (간략)**
- 팀원 API·LLM·음성·프론트엔드 입출력 규격 확인 및 경로 조정
- 병렬 개발 결과 `main` 반영, Docker Compose 실행 순서·프로필 정리
- 동작 흐름 및 운영 문서 정리

**도구 활용**
- Git 이력으로 변경 범위·작성 근거 확인
- Docker Compose로 다중 서비스 기동 순서 검증
- pytest로 API·Bridge·voice-agent·LLM **계약 테스트**
- 시리얼·서비스 로그로 STT·LLM·TTS 단계별 점검

---

# Result — 무엇이 달라졌나

| 지표 | Before | After |
|---|---|---|
| 음성 응답으로 연결된 입력 경로 | **0개** | **3개** |
| LLM이 제안 가능한 런타임 설정 | **0개** | **3개** |

**3개 입력 경로**
RP2040 마이크 음성 · Wemos 거리 센서 선제 반응 · 프론트 수동 입력

**LLM 제어 가능 설정**
`wake_enabled` · `distance_detection_enabled` · `distance_threshold_cm`

→ *센서값 저장 중심* 구조에서 *센서 이벤트·LLM 응답·TTS 재생이 연결된* 구조로 전환

# Result — 기여 범위 (정직하게)

- Git 기본 작성자 기준 커밋: **전체 61개 중 37개**
- 커밋 수는 작업량 **참고 지표**일 뿐, 기여 품질·비율로 단정하지 않음
- 동일인 추정 별도 작성자 계정 존재로 기여율 계산은 제외

> 개별 개발된 API·LLM·음성·센서 서비스를 연결해,
> 사용자 발화와 거리 감지가 **실제 음성 응답으로 이어지는
> 3개 입력 경로의 AIoT 흐름**을 구성.
>
> 특히 LLM이 대화 답변과 **장치 동작 제안을 함께 내놓는
> 응답 계약**과 그 안전 검증 계층을 직접 설계.
